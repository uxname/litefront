## 4. Стратегия мокирования (Test Setup)
Главная причина «мигающих» (flakey) тестов — зависимость от сети и сторонних сервисов (Logto, Auth0, GraphQL-сервер). Мы должны полностью изолировать тесты от внешнего мира.

### 4.1. E2E Тесты (Playwright): Имитация OIDC-сессии
Нельзя заставлять E2E-тест реально вводить логин и пароль в форму OIDC-провайдера (это медленно и может заблокироваться капчей/лимитами).
*   **Механизм мокирования:** Так как библиотека `oidc-client-ts` (настроенная в `src/features/auth/api/oidc-client.ts`) использует `window.localStorage` для хранения сессии, мы будем инжектировать фейковую сессию прямо в LocalStorage браузера перед запуском теста.
*   **Реализация (Fixtures):** Создать утилиту `tests/e2e/utils/auth.ts`, которая:
  1. Формирует JSON-объект, имитирующий структуру `User` из `oidc-client-ts` (содержит `access_token`, `id_token`, `profile: { sub: '123', email: 'test@example.com' }`, `expires_at: <future_date>`).
  2. Вызывает `page.evaluate()` для записи этого JSON в `localStorage` по ключу `oidc.user:<VITE_OIDC_AUTHORITY>:<VITE_OIDC_CLIENT_ID>`.
*   Таким образом, при переходе на `/protected` приложение мгновенно "поверит", что пользователь уже авторизован.

### 4.2. E2E Тесты: Перехват GraphQL запросов

# ДОПОЛНЕНИЕ К ТЕХНИЧЕСКОМУ ЗАДАНИЮ (Часть 1 из 2)
## Корректировка фокуса: удаление избыточности и мертвых зон

При детальном анализе исходников выявлены архитектурные нюансы, которые делают часть первоначального плана неэффективной. Мы смещаем фокус с тестирования «всего подряд» на тестирование реальных сценариев использования (ROI-driven testing).

### ❌ 1. Отмена Сценария 3.1 (Unit-тест `AuthGuard.tsx`)
**Причина:** В текущей реализации проекта компонент `AuthGuard` является **мёртвым кодом** (или опциональной утилитой). Если мы посмотрим в `src/routes/protected/index.tsx`, защита маршрута реализована через `beforeLoad` TanStack Роутера, а не через компонент-обертку. Тестировать неиспользуемый компонент бессмысленно.
**Корректировка:** Мы полностью удаляем Сценарий 3.1. Вместо этого мы концентрируем все усилия на **Сценарии 3.3 (Логика `beforeLoad`)**, так как именно он реально защищает приложение.

### ❌ 2. Отмена Сценария 3.5 (Тестирование Sentry)
**Причина:** Функция `initSentry` в `src/shared/lib/sentry/config.ts` — это просто вызов библиотечного метода с конфигурацией. В ней нет ветвлений бизнес-логики (кроме проверки `.env`). Тестировать её — значит тестировать библиотеку `@sentry/react`, что является антипаттерном.
**Корректировка:** Исключаем этот файл из покрытия в `vite.config.ts` (добавляем в `coverage.exclude`). Это сразу поднимет общий процент покрытия, убрав из него код, который не требует тестов.

### ⚠️ 3. Упрощение Сценариев 2.1 и 2.3 (E2E OAuth Flow)
**Причина:** В изначальном ТЗ (п. 4.1) я предложил инжектировать OIDC-сессию напрямую в `localStorage` (ключ `oidc.user:...`). На практике это **экстремально хрупкий подход** (brittle test). Библиотека `oidc-client-ts` использует сложную криптографию (PKCE, state hashes, nonces). Подделка такого JWT/Session объекта вручную потребует написания огромного количества мок-кода, который сломается при минорном обновлении библиотеки.
**Корректировка:**
Вместо хакинга `localStorage` в E2E, мы будем использовать мокирование на уровне сети (Network Interception в Playwright). Мы перехватим запросы к `VITE_OIDC_AUTHORITY` и подменим конфигурацию (.well-known/openid-configuration) и ответы сервера авторизации, либо будем использовать существующую библиотеку для моков OIDC в Playwright.
*Если это окажется слишком сложным, мы перенесем проверку состояний авторизации исключительно на уровень Vitest, а в Playwright оставим только проверку редиректа.*

### 🔄 4. Уточнение Сценария 3.4 (`ErrorFallback.tsx`)
**Причина:** В прошлом ТЗ я упомянул тестирование функции `detectErrorCategory`. Но эта функция — **приватная** (не экспортируется из файла).
**Корректировка:** Мы не можем импортировать её в тест. Мы будем тестировать её *поведенчески* (Black Box). В тесте мы будем рендерить сам `<ErrorFallback error={new Error("401")} />` и проверять `screen.getByText()`. Это более правильный подход с точки зрения Testing Library.

# ДОПОЛНЕНИЕ К ТЕХНИЧЕСКОМУ ЗАДАНИЮ (Часть 2 из 2)

## 5. Элегантное решение для E2E тестов OIDC (Паттерн "Test Double")

Как мы выяснили в первой части, ручная подделка PKCE-токенов и стейтов в `localStorage` делает тесты крайне хрупкими. Полный прогон через реальный Logto/Auth0 ломает изоляцию и замедляет CI.

**Утвержденная стратегия:** Использование переменной окружения для подмены провайдера (Mock Provider Injection).

1. **Модификация исходного кода (только для тестовой среды):**
   Мы добавим проверку в `src/features/auth/index.ts` или `src/main.tsx`. Если приложение собирается/запускается с флагом `VITE_MOCK_AUTH=true` (который Playwright будет передавать при запуске тестов), реальный `AuthProvider` из `react-oidc-context` подменяется на `MockAuthProvider`.
2. **Как работает `MockAuthProvider`:**
   Он реализует тот же интерфейс `AuthContextProps`, но берет состояние авторизации из `localStorage` (например, флаг `isTestAuthenticated`).
3. **Действия в Playwright:**
   Теперь в тестах `protected.spec.ts` нам достаточно сделать:
   ```typescript
   await page.evaluate(() => localStorage.setItem('isTestAuthenticated', 'true'));
   await page.goto('/protected');
   ```
   Это на 100% стабильно, мгновенно работает и не зависит от сети, сторонних криптографических библиотек или изменений в OIDC-сервере. Тестирование логики редиректов (для неавторизованных) также будет работать идеально.

## 6. Хирургическая настройка Coverage (`vite.config.ts`)

Метрика "Покрытие 80%" не должна достигаться написанием тестов ради тестов (testing boilerplate). Мы исключим из метрик файлы, которые выполняют связующую, конфигурационную или декларативную роль, и оставим только чистую бизнес-логику.

**Необходимо добавить в `test.coverage.exclude` (в `vite.config.ts`):**

```javascript
exclude: [
  // Базовые исключения (уже есть)
  "node_modules/", "src/generated/", "tests/", "**/*.d.ts", "**/*.config.*", 
  
  // 1. Точки входа и глобальные провайдеры (тестируются через E2E, Unit здесь бесполезен)
  "src/main.tsx",
  "src/routes/__root.tsx",
  "src/app/providers/**",
  
  // 2. Конфигурация и адаптеры сторонних библиотек
  "src/shared/lib/sentry/**", // Обсудили в Части 1 (просто вызовы API Sentry)
  "src/shared/config/**",     // Парсинг .env (нет сложной логики)
  "src/features/auth/api/oidc-client.ts", // Просто объект oidcConfig
  
  // 3. Мертвый или унаследованный код
  "src/features/auth/ui/AuthGuard.tsx" // Не используется, логика в beforeLoad роутера
]
```

Очистив этот "шум" из метрик, порог в 80% станет абсолютно честным: он будет требовать покрытия реальных компонентов (`ErrorFallback`, `Counter`), функций (`createGraphQLClient`) и логики роутинга (`beforeLoad`).
